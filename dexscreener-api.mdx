# Dexscreener API

## Base URL

```text
http://127.0.0.1:3002
```

## Upstream source

This local surface wraps the official Dexscreener API documented at:

- [Dexscreener API Reference](https://docs.dexscreener.com/api/reference)

The local wrapper keeps naming more explicit than the upstream path layout and normalizes every response into:

- `ok`
- `fetchedAt`
- `request`
- `status`
- `data`
- `error?`

## Design goals

- keep the wrapper close to the official Dexscreener contract
- split latest lists from market lookup endpoints
- make pair lookups easier to understand than the mixed upstream naming
- keep Swagger and Mintlify stable even if the upstream docs page moves around

## Endpoint map

### Dashboard

| Local endpoint | Purpose | Upstream |
| --- | --- | --- |
| `GET /api/dexscreener/dashboard` | one-call latest dashboard payload | aggregate of the latest list endpoints below |

### Latest lists

These are the low-rate public list endpoints. Dexscreener documents them at `60 requests / minute`.

| Local endpoint | Purpose | Official upstream |
| --- | --- | --- |
| `GET /api/dexscreener/token-profiles/latest` | latest token profile cards | `GET /token-profiles/latest/v1` |
| `GET /api/dexscreener/community-takeovers/latest` | latest community takeover cards | `GET /community-takeovers/latest/v1` |
| `GET /api/dexscreener/ads/latest` | latest paid ads cards | `GET /ads/latest/v1` |
| `GET /api/dexscreener/token-boosts/latest` | newest boost purchases | `GET /token-boosts/latest/v1` |
| `GET /api/dexscreener/token-boosts/top` | top boosted tokens | `GET /token-boosts/top/v1` |

### Market lookups

These are the higher-rate pair and search endpoints. Dexscreener documents them at `300 requests / minute`.

| Local endpoint | Purpose | Official upstream |
| --- | --- | --- |
| `GET /api/dexscreener/search` | free-text pair search | `GET /latest/dex/search/?q=...` |
| `GET /api/dexscreener/pairs/by-pair` | one or more pair addresses | `GET /latest/dex/pairs/{chainId}/{pairAddress}` |
| `GET /api/dexscreener/pairs/by-token` | all pairs for one token address | `GET /token-pairs/v1/{chainId}/{tokenAddress}` |
| `GET /api/dexscreener/pairs/by-tokens` | all pairs for multiple token addresses | `GET /tokens/v1/{chainId}/{tokenAddresses}` |

### Orders

| Local endpoint | Purpose | Official upstream |
| --- | --- | --- |
| `GET /api/dexscreener/orders` | paid orders and boosts for a token | `GET /orders/v1/{chainId}/{tokenAddress}` |

## Local naming rules

The local API intentionally renames the pair endpoints so they are easier to reason about:

- `pairs/by-pair` means lookup by pair address
- `pairs/by-token` means lookup by one token address
- `pairs/by-tokens` means lookup by multiple token addresses

This avoids mixing the upstream `latest/dex/pairs`, `token-pairs`, and `tokens` naming styles in client code.

## Example calls

### Latest token profiles

```bash
curl "http://127.0.0.1:3002/api/dexscreener/token-profiles/latest?limit=10"
```

### Search by symbol pair

```bash
curl "http://127.0.0.1:3002/api/dexscreener/search?q=SOL/USDC&limit=20"
```

### Pair lookup by pair address

```bash
curl "http://127.0.0.1:3002/api/dexscreener/pairs/by-pair?chainId=solana&pairId=Czfq3xZZDmsdGdUyrNLtRhGc47cXcZtLG4crryfu44zE"
```

### Pair lookup by one token

```bash
curl "http://127.0.0.1:3002/api/dexscreener/pairs/by-token?chainId=solana&tokenAddress=So11111111111111111111111111111111111111112&limit=20"
```

### Pair lookup by multiple tokens

```bash
curl "http://127.0.0.1:3002/api/dexscreener/pairs/by-tokens?chainId=solana&tokenAddresses=So11111111111111111111111111111111111111112,JUPyiwrYJFskUPiHa7hkeR8VUtAeFoSYbKedZNsDvCN&limit=30"
```

### Token orders

```bash
curl "http://127.0.0.1:3002/api/dexscreener/orders?chainId=solana&tokenAddress=JUPyiwrYJFskUPiHa7hkeR8VUtAeFoSYbKedZNsDvCN"
```

## Notes

- the local wrapper adds a short-lived cache to avoid burning through Dexscreener rate limits during UI polling
- latest list endpoints are intentionally split from market lookups because they have different upstream rate limits
- `pairs/by-tokens` accepts comma-separated token addresses and keeps the official upstream limit of `30` addresses per request
- if you need the exact response schema, use the autogenerated `Dexscreener` OpenAPI group in the sidebar
